Fundamentos necessários

Um LLM (do inglês Large Language Model, Modelo de Linguagem Grande) não tem estado persistente entre conversas. Cada chamada é stateless: o modelo recebe uma sequência de tokens e produz a próxima. Não existe memória, preferências ou contexto preservado de sessão para sessão — a não ser que esses elementos sejam explicitamente injetados no prompt antes da mensagem do usuário.

Esse é o ponto de partida: tudo que o modelo “sabe” sobre como se comportar em uma sessão precisa estar na janela de contexto. A janela de contexto (context window) é o conjunto de tokens visíveis ao modelo no momento da geração — tipicamente composto de: (1) instruções de sistema, (2) histórico de mensagens, (3) resultados de ferramentas ativadas. A distinção entre instruções, skills e agentes é, em essência, uma distinção sobre o que entra na janela de contexto, quando e com que autonomia o modelo age.

O mecanismo de atenção do Transformer processa todos esses tokens sem hierarquia formal — tecnicamente, uma instrução no início do contexto tem o mesmo peso computacional que uma mensagem do usuário ao final. Na prática, os modelos modernos são treinados com RLHF (do inglês Reinforcement Learning from Human Feedback, aprendizado por reforço com feedback humano) para priorizar instruções de sistema, gerando comportamento de deferência que não está na arquitetura, mas no treinamento. Instruções mal escritas não falham por razões técnicas — falham porque o modelo não consegue interpretar a intenção do autor, ou porque existe ambiguidade entre instrução e pedido do usuário.

Conceitos-chave

System prompt

Texto injetado antes da conversa. Define comportamento, tom, restrições e ferramentas disponíveis para toda a sessão.

Instrução de projeto

Subconjunto do system prompt associado a uma pasta ou projeto. Persiste entre sessões naquele escopo. Controlado pelo usuário.

Skill

Arquivo SKILL.md com instruções especializadas e opcionalmente ferramentas. Ativado por gatilho semântico ou explícito.

Agente

Loop autônomo de planejamento → ação → observação → novo planejamento. Pode usar ferramentas e spawnar sub-agentes.

MCP

Model Context Protocol: padrão que expõe ferramentas externas (Slack, Notion, Bash, APIs) ao modelo via chamadas estruturadas.

Tool use

Capacidade do modelo de emitir chamadas estruturadas a funções externas durante a geração, receber resultados e continuar o raciocínio.

Instruções de sistema — o que são e como funcionam

Contexto histórico O conceito de “system prompt” emergiu com os primeiros modelos de chat fine-tuned (GPT-3.5-turbo, 2022). Antes disso, o prompt era simplesmente texto livre sem separação de papéis. A divisão de turnos (system / user / assistant) foi introduzida na API da OpenAI como convenção, adotada rapidamente pela Anthropic e demais provedores. O treinamento com RLHF para respeitar a instrução de sistema — em vez de apenas completar o texto — é o que tornou essa separação funcionalmente significativa. Sem esse treinamento, o modelo trataria o system prompt como qualquer outro parágrafo.

Uma instrução de sistema é texto injetado no início da janela de contexto, antes do primeiro turno do usuário. Ela define o “contrato” de comportamento do modelo para toda a sessão. Há quatro camadas onde instruções podem ser fornecidas, em ordem decrescente de precedência prática:

  1. System prompt da aplicação — injetado pelo desenvolvedor via API. O usuário não vê e normalmente não pode sobrescrever. Define regras fundamentais de segurança, tom e escopo da ferramenta.
  2. Instruções de projeto — no Cowork, texto associado a uma pasta ou projeto. Persiste entre sessões naquele escopo. O usuário controla esse nível.
  3. Memória persistida — fatos sobre o usuário e preferências salvos explicitamente pelo modelo em arquivos e reinjetados nas sessões seguintes. Complementa as instruções com dados dinâmicos.
  4. Instruções na mensagem — o usuário escreve o contexto diretamente no prompt. O mais frágil: aplica-se apenas à conversa atual e depende de o usuário lembrar de incluir cada vez.

O mecanismo pelo qual a instrução opera é simples: os tokens da instrução aparecem antes dos tokens do usuário. O modelo foi treinado para produzir respostas coerentes com esse contexto precedente. “Coerência” aqui é aprendida estatisticamente — o modelo viu durante o treinamento inúmeros exemplos de sistemas que respeitam instruções — e é reforçada via RLHF. Não existe flag especial que diz “isso é uma instrução, obedeça”; existe apenas a posição no contexto e o treinamento que ensinou o modelo a deferenciar a ela.

Limites do modelo

Instruções de sistema não são invioláveis. Um usuário experiente pode usar técnicas de prompt injection (injeção de prompt) para sobrescrever partes da instrução — especialmente em aplicações que permitem ao usuário injetar texto processado como parte do contexto. Além disso, instruções muito longas ou contraditórias sofrem com atenção diluída: o modelo pode não seguir instruções posicionadas no meio do system prompt se o contexto estiver muito carregado. O treinamento RLHF mitiga, mas não elimina, essas vulnerabilidades.

Como escrever boas instruções

Escrever uma instrução eficaz é essencialmente um problema de especificação: você precisa comunicar ao modelo não apenas o que fazer, mas o que fazer em casos que você não antecipou. A maioria das instruções ruins falha porque foi escrita para o caso esperado e silencia nos casos de borda.

Princípio 1 — Comece pelo papel, não pela lista de regras

O modelo generaliza melhor a partir de um papel bem definido do que de uma lista de regras. “Você é um assistente de pesquisa especializado em direito tributário brasileiro, trabalhando para um escritório de advocacia. Seu interlocutor é sempre um advogado sênior, nunca um cliente.” — essa descrição contextualiza automaticamente tom, vocabulário, profundidade técnica e o que omitir, sem precisar listar cada caso. Regras explícitas são necessárias apenas onde o papel genérico falharia.

Princípio 2 — Especifique o formato de saída com exemplos, não só com descrições

Descrições de formato (“responda de forma concisa e estruturada”) são interpretadas de forma diferente pelo modelo em diferentes contextos. Exemplos são inequívocos. Se você quer que o modelo sempre cite a fonte antes da resposta, mostre um exemplo de como isso se parece. Se quer JSON, mostre o schema. O modelo não adivinha intenção — ele generaliza a partir do padrão mais próximo que viu.

# Ruim
Sempre inclua fontes nas respostas.

# Melhor
Ao mencionar qualquer dado ou afirmação, cite a fonte inline:
"A taxa Selic está em 10,5% ao ano (Banco Central do Brasil, jun. 2026)."
Se não houver fonte verificável, diga explicitamente:
"Não tenho fonte confiável para essa afirmação."

Princípio 3 — Diga o que fazer, não só o que não fazer

Proibições (“não faça X”) são menos eficazes que instruções positivas (“quando Y acontecer, faça Z em vez de X”). O motivo é mecanicista: uma proibição suprime uma ação, mas não diz ao modelo qual ação tomar no lugar. Em distribuições de tokens onde X seria a resposta mais provável, a proibição reduz a probabilidade de X mas distribui essa massa de probabilidade entre outras opções — que podem incluir outros comportamentos indesejados. A instrução positiva canaliza a distribuição para o comportamento que você quer.

Princípio 4 — Seja explícito sobre os casos de borda

Pergunte-se: “o que o modelo deve fazer quando o usuário pedir algo fora do escopo?” Se você não especificar, o modelo vai inferir — e a inferência pode não corresponder ao que você quer. Casos de borda a cobrir: pedidos fora do escopo definido, informações que o modelo não tem acesso, contradições entre instrução e pedido do usuário, idioma diferente do esperado, ambiguidade no pedido.

Princípio 5 — Teste contra a instrução, não contra a sua intuição

Depois de escrever a instrução, leia-a como se você fosse o modelo: sem contexto externo, sem intenção implícita, processando apenas os tokens que estão ali. Perguntas para se fazer: “Esta instrução é ambígua em algum ponto? Existe conflito entre dois parágrafos? O tom do exemplo contradiz a instrução de tom?” A maioria dos problemas com instruções são problemas de consistência interna que o autor não vê porque sabe o que queria dizer.

Princípio 6 — Estruture com hierarquia, não com comprimento

Instruções longas não são instruções melhores. Um system prompt de 3000 tokens sem estrutura é menos eficaz que um de 500 tokens bem organizado. Use seções com títulos claros. Coloque as informações mais importantes no início e no final — os modelos demonstram empiricamente maior aderência a instruções nas posições extremas da janela de contexto (efeito de primazia e recência). O meio do system prompt é o lugar de maior risco para instruções críticas.

Princípio 7 — Separe instruções de contexto factual

Uma boa instrução tem dois tipos de conteúdo: (a) instruções comportamentais — o que fazer — e (b) contexto factual — o que é verdade sobre o ambiente. Misturá-los cria confusão. Separe em seções distintas: ## Contexto e ## Comportamento. Isso facilita a manutenção da instrução ao longo do tempo e reduz o risco de o modelo tratar contexto como instrução e vice-versa.

Controvérsia

Há debate sobre se o comprimento do system prompt ajuda ou atrapalha. Alguns praticantes relatam que prompts muito detalhados fazem o modelo “seguir o script” mesmo quando o usuário fornece contexto que justificaria adaptação — resultando em respostas mecanicamente corretas mas contextualmente inadequadas. Outros argumentam que prompts detalhados aumentam consistência e reduzem variância. A evidência empírica é mista e depende muito do modelo e da tarefa. Posição defensável: use o mínimo necessário para cobrir os casos de borda críticos, deixe o modelo generalizar o resto.

Skills — instruções especializadas sob demanda

Contexto histórico O conceito de skills como módulos ativáveis emergiu da necessidade de evitar system prompts monolíticos. Um único prompt tentando cobrir todos os casos de uso de uma plataforma fica inconsistente e longo demais para ser eficaz. Skills resolvem isso com composição: um núcleo de instruções gerais + módulos específicos ativados conforme o contexto. No Cowork (Anthropic, 2025), skills são arquivos SKILL.md que o modelo lê dinamicamente quando o contexto semântico ou uma instrução explícita indica que aquele módulo é relevante.

Uma skill é tecnicamente um arquivo de texto com instruções especializadas e, opcionalmente, declarações de ferramentas. O mecanismo de ativação funciona assim: o modelo recebe na janela de contexto uma lista de skills disponíveis com suas descrições. Quando o usuário faz um pedido, o modelo avalia semanticamente se alguma skill é relevante — se for, lê o arquivo SKILL.md correspondente e o incorpora ao contexto antes de responder. Isso é essencialmente retrieval (recuperação) seletivo de instruções: o system prompt geral fica enxuto, e as instruções especializadas entram apenas quando necessário.

A anatomia de uma skill tem três partes:

  1. Descrição de ativação — texto que aparece na lista de skills disponíveis. O modelo usa esse texto para decidir se a skill é relevante. Uma descrição ruim resulta em ativação incorreta: skill ativada quando não deveria, ou não ativada quando deveria.
  2. Instruções de execução — o conteúdo do SKILL.md. Descreve passo a passo o que o modelo deve fazer ao executar aquela skill, quais ferramentas usar, qual formato de saída produzir.
  3. Declaração de ferramentas (opcional) — quais MCPs ou funções externas a skill precisa. Sem essa declaração, a skill opera apenas sobre tokens, sem acesso a arquivos, APIs ou outros sistemas externos.

Como ativar uma skill

Há três formas de ativação. A primeira é semântica: o modelo infere que a skill é relevante com base no pedido. “Crie uma apresentação sobre X” ativa automaticamente a skill de PPTX se sua descrição diz “Use when creating slide decks or .pptx files”. A segunda é explícita: o usuário ou o system prompt instrui o modelo a usar uma skill específica por nome. A terceira é por gatilho declarado: a própria descrição da skill lista termos que devem disparar ativação — “MANDATORY TRIGGERS: PDF, .pdf, form, extract, merge”. Gatilhos explícitos reduzem ambiguidade semântica mas são menos flexíveis a variações na linguagem do usuário.

Escrevendo uma skill eficaz

A descrição de ativação deve ser específica o suficiente para evitar falsos positivos e abrangente o suficiente para cobrir variações legítimas do pedido. Uma boa heurística: escreva a descrição como se você estivesse explicando a um colega quando chamar aquela skill — não o que ela faz, mas quando ativá-la. “Use quando o usuário mencionar um arquivo .xlsx ou pedir para criar uma planilha” é mais preciso que “Use para tarefas relacionadas a dados”.

O conteúdo do SKILL.md deve ser orientado a processo, não a resultado. Em vez de “produza um documento Word de alta qualidade”, escreva “1. Instale a dependência com pip; 2. Crie o arquivo na pasta do projeto; 3. Verifique a integridade; 4. Apresente ao usuário com present_files”. O modelo segue melhor instruções sequenciais concretas do que objetivos abstratos.

## Quando usar
Ative esta skill para qualquer pedido envolvendo arquivo .docx:
criação, leitura, edição ou conversão.

## Passos de execução
1. Verificar se python-docx está instalado:
   pip show python-docx
   Se ausente: pip install python-docx --break-system-packages

2. Ler o conteúdo ou estrutura solicitada pelo usuário

3. Gerar o arquivo no diretório do projeto (nunca fora dele)

4. Verificar integridade:
   python -c "from docx import Document; Document('saida.docx')"

5. Apresentar ao usuário via ferramenta present_files

## Restrições
- Nunca criar arquivo fora da pasta mapeada do projeto
- Sempre salvar em UTF-8
Limites do modelo

Skills dependem de boa semântica na descrição de ativação. Se duas skills têm descrições sobrepostas, o modelo pode ativar a errada — ou tentar usar ambas, gerando conflito de instruções. A solução é garantir que as descrições sejam mutuamente exclusivas nos casos de borda. Outra limitação: skills com muitos passos em sequência são vulneráveis a erros de execução parcial — o modelo pode pular etapas se o contexto estiver muito carregado. Para skills complexas, dividir em sub-skills menores é mais robusto do que uma skill monolítica.

Agentes — autonomia além de uma resposta

Contexto histórico O paradigma de agentes com LLMs foi popularizado pelo paper ReAct (YAO et al., 2022) e pela implementação AutoGPT (2023), que demonstrou — de forma espetacular e caótica — que modelos podiam encadear múltiplas ações autonomamente. O problema era que loops sem restrição falhavam de formas imprevisíveis e custosas. A evolução posterior (Claude Agent SDK, OpenAI Agents SDK, LangChain Agents) focou em estruturar o loop de agência com: limites de iteração, ferramentas tipadas, mecanismos de handoff e supervisão humana em pontos críticos.

Um agente é um sistema em que o modelo opera em loop: planeja uma ação, executa, observa o resultado, planeja a próxima. A diferença fundamental em relação a uma única resposta é que o agente pode executar múltiplas chamadas de ferramentas em sequência, adaptar o plano com base nos resultados intermediários e — em arquiteturas multi-agente — spawnar (criar e delegar para) sub-agentes especializados.

O loop de um agente tem quatro componentes:

  1. Planejamento — o modelo raciocina sobre a tarefa e decide qual ação tomar a seguir. Em modelos com extended thinking (raciocínio estendido), essa etapa envolve um processo de pensamento interno antes de emitir a chamada de ferramenta.
  2. Ação — o modelo emite uma chamada estruturada de ferramenta (tool call). A ferramenta pode ser: executar código bash, ler um arquivo, buscar na web, criar um documento, chamar uma API externa via MCP.
  3. Observação — o resultado da ferramenta retorna ao modelo como parte do contexto. O modelo “vê” o resultado antes de decidir a próxima ação.
  4. Iteração — o ciclo se repete até que o modelo julgue a tarefa completa ou até um limite externo de iterações ser atingido.

Tipos de arquitetura de agentes

Agente único com ferramentas — um modelo opera com acesso a um conjunto de ferramentas. Adequado para tarefas sequenciais onde a ordem das ações é previsível. Exemplo: pesquisar na web, extrair dados, formatar e salvar um relatório.

Orquestrador + sub-agentes — um agente coordenador decompõe a tarefa e spawna agentes especializados para subtarefas paralelas. Os sub-agentes retornam resultados ao orquestrador, que sintetiza e decide próximos passos. Adequado para tarefas paralelizáveis. O custo é maior latência de setup e maior risco de erros de coordenação.

Pipeline de agentes — saída de um agente é entrada do próximo. Adequado para workflows com etapas bem definidas onde cada etapa tem domínio especializado. Exemplo: agente extrator → agente validador → agente formatador. O risco é propagação de erros: um erro na etapa 1 contamina todas as seguintes.

Quando criar um agente

A decisão de usar um agente — em vez de uma instrução bem escrita ou uma skill — deve ser guiada pela natureza da tarefa. Agentes têm custo (tokens por iteração, latência, risco de erros em cascata) que só se justifica quando a tarefa tem estas características: o resultado de uma etapa determina o que fazer na seguinte (sequência adaptativa); o escopo não é conhecido antecipadamente; múltiplas ferramentas precisam ser orquestradas em ordem não-trivial; ou a tarefa pode ser decomposta em sub-tarefas paralelizáveis.

Escrevendo instruções para agentes

Agentes precisam de instruções mais detalhadas que respostas simples, porque o modelo toma decisões sem supervisão humana a cada passo. Três elementos são críticos:

Critério de parada: quando o agente deve considerar a tarefa completa? Sem isso, o modelo pode continuar iterando por insegurança — ou parar cedo demais. “Encerre quando tiver encontrado ao menos 3 fontes primárias confirmando o dado” é um critério operacional. “Pesquise até ter informação suficiente” não é.

Estratégia de erro: o que fazer quando uma ferramenta falha? O agente deve tentar novamente? Tentar uma alternativa? Parar e reportar ao usuário? Sem essa instrução, o comportamento padrão é tentativa de recuperação implícita — que pode incluir ações indesejadas ou loop de retentativas.

Escopo de autonomia: o que o agente pode fazer sem confirmação do usuário? Operações destrutivas (deletar arquivos, enviar e-mails, fazer deploys, executar transações) devem requerer confirmação explícita. O agente precisa saber onde está o limite entre “pode agir” e “deve perguntar”.

Limites do modelo

Agentes acumulam erros: um erro na etapa 2 pode invalidar todas as etapas subsequentes sem que o modelo detecte. Esse problema — chamado de error compounding (acúmulo de erros) — é mais grave em pipelines longos. Mitigação: incluir checkpoints de verificação no loop (o agente verifica o resultado de etapas críticas antes de prosseguir). Agentes também podem entrar em loops: tentar a mesma ação que falhou repetidamente sem perceber que está repetindo. Limite de iterações externo é obrigatório em qualquer implementação de produção.

Quando usar cada um — o mapa de decisão

A escolha entre instrução, skill e agente não é estética — tem implicações diretas em custo, confiabilidade e adequação à tarefa.

Dimensão Instrução de sistema Skill Agente
Ativação Sempre ativa (toda sessão) Sob demanda, por contexto semântico Explicitamente invocado para uma tarefa
Escopo Comportamento geral e persistente Domínio específico (tipo de arquivo, ferramenta) Tarefa específica com múltiplos passos
Autonomia Nenhuma — define como responder Baixa — define como executar um tipo de tarefa Alta — decide a sequência de ações
Custo Tokens fixos por sessão Tokens extras somente quando ativada Tokens por iteração × número de iterações

A heurística prática: comece com a instrução de sistema mais simples que resolve o problema. Se você precisar de comportamentos especializados para tipos de tarefa específicos, adicione skills. Se a tarefa requerer múltiplos passos adaptativos com ferramentas, use um agente — mas defina explicitamente o escopo de autonomia e os critérios de parada antes de deixar o loop rodar.

Implementação

O código abaixo demonstra os três mecanismos em um exemplo mínimo usando a API da Anthropic. O objetivo não é cobrir todos os casos — é tornar concreto o que, até aqui, foi descrito em prosa: como instrução, skill e agente diferem no nível de chamada de API.

import anthropic
import json

client = anthropic.Anthropic()  # usa ANTHROPIC_API_KEY do ambiente

# ─────────────────────────────────────────────
# 1. INSTRUÇÃO DE SISTEMA — define comportamento persistente
# ─────────────────────────────────────────────
SYSTEM_PROMPT = """
## Papel
Você é um assistente de análise financeira pessoal.
Seu interlocutor é sempre Vinicius, investidor pessoa física.

## Comportamento
- Responda sempre em português formal, sem gírias
- Cite fontes quando mencionar dados de mercado
- Ao receber pedidos fora do escopo financeiro, redirecione gentilmente

## Fora do escopo
Não execute código, não acesse arquivos, não faça pesquisas na web.
""".strip()

def resposta_simples(pergunta: str) -> str:
    """Usa apenas a instrução de sistema — sem ferramentas."""
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=SYSTEM_PROMPT,
        messages=[{"role": "user", "content": pergunta}]
    )
    return msg.content[0].text


# ─────────────────────────────────────────────
# 2. SKILL — instrução especializada injetada dinamicamente
# ─────────────────────────────────────────────
SKILL_CALCULAR_RETORNO = """
## Skill: Cálculo de Retorno Ajustado

Quando ativada, execute EXATAMENTE estes passos:
1. Extraia da mensagem: valor_inicial, valor_final, periodo_meses
2. Calcule retorno_total = (valor_final / valor_inicial - 1) * 100
3. Calcule retorno_mensal = ((valor_final / valor_inicial) ** (1/periodo_meses) - 1) * 100
4. Calcule retorno_anual = ((1 + retorno_mensal/100) ** 12 - 1) * 100
5. Responda com tabela: | Métrica | Valor |

Se faltarem dados, peça apenas o que está faltando.
""".strip()

def resposta_com_skill(pergunta: str, skill_ativa: bool = False) -> str:
    """Injeta a skill no system prompt quando relevante."""
    system = SYSTEM_PROMPT
    if skill_ativa:
        # A skill entra como seção adicional do system prompt
        system += f"\n\n---\n{SKILL_CALCULAR_RETORNO}"

    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": pergunta}]
    )
    return msg.content[0].text


# ─────────────────────────────────────────────
# 3. AGENTE — loop autônomo com tool use
# ─────────────────────────────────────────────

                "ticker": {"type": "string", "description": "Ex: PETR4, VALE3, BTC-USD"}

        }

    {
        "name": "calcular_alocacao",

            "type": "object",

                        "properties": {
                            "ativo": {"type": "string"},
                            "peso": {"type": "number", "description": "Percentual, ex: 30 para 30%"}
                        }
                    }
                }
            },
            "required": ["valor_total", "alocacoes"]
        }
    }
]

def executar_ferramenta(nome: str, inputs: dict) -> str:
    """Simula a execução das ferramentas — substituir por chamadas reais."""
    if nome == "buscar_cotacao":
        cotacoes_mock = {"PETR4": 38.50, "VALE3": 62.30, "BTC-USD": 102000.00}
        preco = cotacoes_mock.get(inputs["ticker"], 0)
        return json.dumps({"ticker": inputs["ticker"], "preco": preco, "moeda": "BRL"})

    if nome == "calcular_alocacao":
        total = inputs["valor_total"]
        resultado = []
        for item in inputs["alocacoes"]:
            valor = total * item["peso"] / 100
            resultado.append({"ativo": item["ativo"], "valor": round(valor, 2)})
        return json.dumps({"alocacoes": resultado})

    return json.dumps({"erro": f"Ferramenta {nome} não encontrada"})


def agente_financeiro(tarefa: str, max_iter: int = 5) -> str:
    """
    Loop de agente: planejamento → ação → observação → iteração.
    max_iter evita loops infinitos — critério de parada externo obrigatório.
    """
    mensagens = [{"role": "user", "content": tarefa}]

    for iteracao in range(max_iter):
        resposta = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            system=SYSTEM_PROMPT,
            tools=FERRAMENTAS,
            messages=mensagens
        )

        # Critério de parada: modelo decide que terminou (stop_reason = "end_turn")
        if resposta.stop_reason == "end_turn":
            # Extrai texto da resposta final
            for bloco in resposta.content:
                if hasattr(bloco, "text"):
                    return bloco.text
            return "Tarefa concluída sem resposta de texto."

        # O modelo quer usar uma ferramenta (stop_reason = "tool_use")
        if resposta.stop_reason == "tool_use":
            # Adiciona a resposta do modelo ao histórico
            mensagens.append({"role": "assistant", "content": resposta.content})

            # Executa cada ferramenta solicitada e coleta resultados
            resultados = []
            for bloco in resposta.content:
                if bloco.type == "tool_use":
                    resultado = executar_ferramenta(bloco.name, bloco.input)
                    resultados.append({
                        "type": "tool_result",
                        "tool_use_id": bloco.id,
                        "content": resultado
                    })

            # Devolve os resultados ao modelo (observação do loop)
            mensagens.append({"role": "user", "content": resultados})
            # O loop continua: modelo agora planeja com base nos resultados

    return f"Limite de {max_iter} iterações atingido sem conclusão."

O código ilustra os três níveis de forma comparável. Na instrução pura, tudo o que o modelo “sabe” vem do system prompt — sem ferramentas, sem loop. Na skill, uma seção extra entra no system prompt dinamicamente, especializando o comportamento para o tipo de tarefa. No agente, o modelo emite tool calls, recebe resultados, e decide se precisa de mais ações — o loop continua até que o modelo declare conclusão ou o limite de iterações seja atingido.

A fronteira entre skills e agentes é mais tênue do que parece. Uma skill com muitos passos e chamadas de ferramentas começa a se comportar como um agente limitado. A distinção prática é de intenção de design: uma skill é um módulo especializado com escopo definido, ativado e concluído em uma única resposta. Um agente é um sistema de múltiplos turnos com estado interno distribuído nas mensagens do histórico. Quando uma skill precisa de mais de um ciclo de ferramenta, a decisão arquitetural é: promovê-la a agente, ou quebrar em etapas que o usuário conduz manualmente.

Perguntas em aberto

Questão aberta 1

Qual é o tamanho ótimo de um system prompt? Há evidência empírica de que prompts mais longos reduzem aderência a instruções específicas (efeito de atenção diluída), mas a magnitude desse efeito varia por modelo e por tarefa. Não existe, até o momento, uma teoria quantitativa que permita prever o ponto de inflexão onde comprimento começa a prejudicar em vez de ajudar.

Questão aberta 2

Como escalar agentes multi-hop confiáveis? O problema de error compounding em pipelines longos ainda não tem solução geral. Técnicas como checkpoints de verificação e sub-agentes especializados reduzem o problema, mas não o eliminam. Há debate ativo sobre se a solução é arquitetural (modelos melhores em detectar próprios erros) ou de design de sistema (loops de verificação externos obrigatórios).

Questão aberta 3

Qual é o limite da especificação via linguagem natural? Instruções em prosa são ambíguas por natureza — a língua natural carrega ambiguidade semântica irredutível. Há pesquisa crescente sobre linguagens de especificação mais formais para comportamento de modelos (DSLs para prompts, schemas de comportamento tipados). A questão em aberto é se formalidade adicional compensa o custo de adoção e reduz expressividade.

Questão aberta 4

Como garantir segurança em agentes com escopo de autonomia amplo? À medida que agentes ganham acesso a mais ferramentas e operam em loops mais longos, o risco de ações não-intencionais com consequências reais (envio de e-mails, deleção de dados, execução de transações) aumenta. O design de “human-in-the-loop” em pontos críticos é consensual, mas quais são os pontos certos — e como o modelo sabe identificá-los — é uma questão em aberto.

Discussão

A pergunta motivadora era: por que cada conversa começa do zero e como instruções, skills e agentes resolvem esse problema de formas diferentes? A resposta emergiu ao longo do compêndio em três camadas.

O início do zero é uma consequência da natureza stateless dos LLMs: sem injeção explícita de contexto, o modelo não tem memória. Instruções de sistema resolvem a camada de comportamento — o “como trabalhar” — tornando-o persistente entre sessões. Skills resolvem a camada de especialização — o “como executar este tipo de tarefa” — sem sobrecarregar o system prompt com instruções que raramente são necessárias. Agentes resolvem a camada de execução — o “faça isso sem precisar me guiar em cada passo” — ao custo de maior autonomia e, portanto, maior risco.

A tensão central do campo é entre especificidade e generalidade. Instruções muito específicas criam modelos previsíveis mas frágeis — funcionam bem no caso esperado e falham de forma estranha nos casos de borda. Instruções muito gerais criam modelos flexíveis mas inconsistentes. O ideal — instruções gerais com exemplos concretos para casos de borda críticos — é difícil de manter à medida que o sistema cresce. É por isso que skills existem: elas são a resposta arquitetural ao problema de escalar especificidade sem escalar o custo de atenção.

Uma implicação que não foi enunciada no corpo do compêndio: a qualidade de uma instrução é inversamente proporcional ao quanto o usuário precisa pensar sobre ela depois de escrita. Uma boa instrução funciona em silêncio — o usuário só percebe que existe quando ela falta. Isso sugere um critério de avaliação prático: se você está frequentemente adicionando contexto no início de cada conversa para corrigir o comportamento do modelo, sua instrução de sistema está incompleta. Se você está frequentemente corrigindo o modelo durante a conversa, sua instrução de sistema está ambígua. Se o modelo raramente precisa de correção, a instrução está bem calibrada.
