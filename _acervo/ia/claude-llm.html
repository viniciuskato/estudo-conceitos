O que é o Claude

Nota histórica A Anthropic foi fundada em 2021 por ex-pesquisadores da OpenAI — incluindo Dario Amodei, Daniela Amodei e Chris Olah — com foco explícito em segurança de IA. O primeiro modelo Claude foi lançado em março de 2023. A empresa introduziu o conceito de Constitutional AI (IA Constitucional) como alternativa ao RLHF puro: em vez de feedback humano para cada comportamento, um conjunto de princípios escritos guia o refinamento do modelo. O nome Claude é uma homenagem a Claude Shannon, fundador da teoria da informação.

Claude é um large language model (LLM — modelo de linguagem de grande escala) desenvolvido pela Anthropic. Na prática, é um sistema de predição de tokens treinado para ser útil, inofensivo e honesto. A distinção da Anthropic em relação a concorrentes está no método de alinhamento: além do reinforcement learning from human feedback (RLHF — aprendizado por reforço com feedback humano), usam Constitutional AI (IA Constitucional), onde o modelo aprende a avaliar suas próprias respostas com base em princípios explícitos, gerando dados sintéticos de preferência.

Você interage com o Claude através do Cowork — uma interface de área de trabalho construída sobre o Claude Agent SDK, que adiciona acesso a arquivos, shell Linux isolado, memória persistente entre sessões e capacidade de invocar ferramentas externas (MCPs). O modelo subjacente na sua sessão atual é o Claude Sonnet 4.6, posicionado como o equilíbrio ótimo entre capacidade e velocidade na família Claude 4.

Definição formal

LLM: modelo probabilístico que mapeia uma sequência de tokens de entrada para uma distribuição de probabilidade sobre o próximo token, treinado em corpus massivo de texto para capturar estrutura estatística da linguagem. A geração de texto é a aplicação iterativa desse processo de predição — o modelo não "pensa" e depois escreve; ele gera token por token, e o processo de geração é o raciocínio.

Fundamentos necessários

Este compêndio não pressupõe conhecimento técnico de machine learning. Os conceitos necessários são introduzidos conforme aparecem. Familiaridade com uso básico do Claude (enviar mensagens, receber respostas) é suficiente como ponto de partida.

Conceitos-chave

Token

Unidade básica de texto — aproximadamente ¾ de palavra em inglês, ~½ palavra em português. Limite de contexto medido em tokens.

Janela de contexto

Toda a conversa visível ao modelo no momento da geração. Tudo fora da janela não existe para o Claude.

Temperatura

Parâmetro que controla aleatoriedade da geração. Valor baixo = mais determinístico; valor alto = mais criativo e variado.

MCP

Model Context Protocol — padrão aberto para conectar o LLM a ferramentas externas: arquivos, bancos de dados, APIs, apps.

System prompt

Instrução que antecede a conversa, invisível ao usuário padrão, que define persona, regras e ferramentas disponíveis ao modelo.

Alinhamento

Conjunto de técnicas para que o modelo siga intenções humanas sem causar dano — o problema central de segurança em LLMs.

Agente

LLM com acesso a ferramentas e capacidade de planejar ações em múltiplos passos — vai além de responder para executar tarefas.

Skill

Arquivo SKILL.md que instrui o Claude a executar um fluxo especializado — ex: criar .docx, .pptx, fazer autocrítica.

Constitutional AI

Método da Anthropic: modelo avalia suas respostas com base em princípios escritos, gerando dados de preferência sem feedback humano por item.

Arquitetura e treinamento

Nota histórica — Transformer O artigo "Attention Is All You Need" (Vaswani et al., 2017) propôs a arquitetura Transformer, que substituiu redes recorrentes pelo mecanismo de autoatenção. A atenção permite ao modelo ponderar a relevância de qualquer posição anterior no texto para prever o próximo token — independentemente da distância. Todos os grandes LLMs modernos (GPT, Gemini, Claude, Llama) são baseados nessa arquitetura ou em variantes dela.

Claude é construído sobre a arquitetura Transformer decoder — o mesmo tipo base de todos os grandes LLMs. O modelo processa o texto de entrada (chamado de prompt) e gera saída token por token, cada token condicionado a todos os anteriores. Internamente, o texto é primeiro convertido em vetores numéricos (embeddings — representações vetoriais), processados por dezenas de camadas de atenção e redes neurais, e então convertidos de volta em tokens de saída.

O que acontece quando você envia uma mensagem

O fluxo real é mais rico do que parece na interface:

  1. Construção do contexto: o Cowork monta o system prompt (suas instruções de projeto, regras de comportamento, memória persistente) + histórico da conversa + sua mensagem atual. Tudo isso é concatenado em uma única sequência de tokens.
  2. Prefill: o modelo processa toda essa sequência de entrada de uma vez — esta fase é rápida porque os tokens de entrada são processados em paralelo.
  3. Geração: o modelo produz tokens de saída um por um. Cada token gerado é adicionado ao contexto antes de gerar o próximo. É por isso que modelos parecem "pensar em tempo real" — eles realmente constroem a resposta sequencialmente.
  4. Chamadas de ferramentas: quando o modelo decide usar uma ferramenta (ler arquivo, executar bash, buscar na web), ele emite um token especial que interrompe a geração, executa a ferramenta, recebe o resultado, e retoma a geração com esse resultado no contexto.

Treinamento em três fases

O Claude passa por três fases distintas antes de chegar até você:

Fase 1 — Pré-treinamento

O modelo aprende a prever tokens em corpus massivo de texto da internet, livros e código — sem supervisão. Nessa fase, o modelo adquire conhecimento factual, capacidade linguística e raciocínio emergente. O modelo resultante (base model) é poderoso, mas sem direcionamento: seguiria qualquer texto, incluindo conteúdo nocivo.

Fase 2 — Supervised Fine-tuning (SFT)

O base model é treinado em demonstrações de comportamento desejado escritas por humanos — pares (pergunta, resposta ideal). Isso orienta o modelo para o formato de assistente conversacional.

Fase 3 — RLHF + Constitutional AI

Humanos ranqueiam pares de respostas por preferência. Um modelo separado (reward model — modelo de recompensa) aprende a prever essas preferências. O Claude é então otimizado para maximizar as recompensas desse modelo via Proximal Policy Optimization (PPO). A Anthropic adiciona a etapa de Constitutional AI: o modelo gera pares de respostas, avalia qual melhor segue os princípios constitucionais, e esses julgamentos sintéticos aumentam os dados de preferência humana.

Limites do modelo de treinamento

O conhecimento do Claude tem data de corte — nada de eventos após o treinamento existe sem ferramentas externas. O modelo não "sabe" o que não viu no treinamento: pode alucinar (gerar texto plausível mas falso) ao tentar responder sobre fatos específicos fora do corpus. O RLHF otimiza para parecer útil e inofensivo aos olhos dos avaliadores — não necessariamente para ser factualmente correto. Isso cria pressão sutil para respostas que soam confiantes mesmo quando incertas.

Como o Claude processa linguagem

O mecanismo de atenção é a peça central. Para cada token que precisa gerar, o modelo calcula uma pontuação de relevância (attention score — pontuação de atenção) para cada token anterior no contexto. Tokens de alta relevância contribuem mais para a representação atual. Com dezenas de "cabeças de atenção" paralelas, o modelo pode simultaneamente capturar relações de longa distância, padrões sintáticos e associações semânticas.

Isso explica um comportamento que você já percebeu: o Claude é sensível a como a pergunta é formulada. Reformular a mesma questão com palavras diferentes pode ativar diferentes associações no espaço de atenção e produzir respostas qualitativamente distintas. Não é arbitrariedade — é estrutura estatística do treinamento refletida na atenção.

Raciocínio emergente e chain-of-thought

LLMs de escala suficiente exibem raciocínio multi-passo — capacidade que não foi explicitamente treinada, mas emergiu do treinamento em texto humano que inclui raciocínio. A técnica de chain-of-thought prompting (raciocínio em cadeia) explora isso: pedir ao modelo para "pensar passo a passo" antes de responder melhora significativamente a acurácia em problemas matemáticos e lógicos. O Claude Sonnet 4.6 usa raciocínio estendido automaticamente em tarefas complexas — você vê o efeito quando a resposta chega em pedaços com etapas intermediárias.

Controvérsia — O Claude realmente "raciocina"?

Posição 1 (emergentista): o raciocínio do Claude é genuíno — os processos internos de atenção implementam algo funcionalmente equivalente a inferência, e a escala produziu capacidades cognitivas emergentes que transcendem a interpolação estatística.

Posição 2 (interpolacionista): o Claude é um sofisticado interpolador estatístico. O que parece raciocínio é correspondência de padrões de alta ordem em espaço de representação vetorial. "Pensar passo a passo" funciona porque o corpus de treinamento está cheio de soluções humanas que usam esse formato — o modelo imita a estrutura, não o processo.

Estado atual: sem consenso. A interpretabilidade mecanicista (campo liderado por Chris Olah, cofundador da Anthropic) está mapeando circuitos internos — descobriu estruturas que se comportam como detectores de negação, emoções funcionais, e planejamento futuro. Se isso constitui "raciocínio real" é uma questão filosoficamente aberta.

Janela de contexto — o recurso mais crítico

A janela de contexto é a memória de trabalho do Claude: tudo que está dentro dela influencia a resposta; tudo fora não existe. O Claude Sonnet 4.6 suporta até 200.000 tokens de contexto — aproximadamente 150.000 palavras ou ~500 páginas de texto. Isso é enorme, mas não ilimitado.

À medida que a conversa cresce, dois fenômenos ocorrem: (1) o custo computacional por resposta aumenta (mais tokens para processar no prefill); (2) a qualidade de atenção sobre eventos distantes no início do contexto pode degradar — o modelo tende a "dar mais peso" ao que está próximo temporalmente da posição atual de geração. É por isso que as instruções de projeto recomendam trocar de sessão para tarefas novas: não é burocracia — é gerenciamento de janela de contexto.

Limites do modelo de atenção

Contextos muito longos sofrem de lost in the middle (perdido no meio): informações no início ou no fim do contexto são mais influentes do que informações no meio. Isso foi documentado em LLMs em geral (LIU et al., 2023) e é parcialmente mitigado em modelos recentes, mas não eliminado. Implicação prática: em sessões muito longas, instruções dadas no começo da conversa têm menos peso do que instruções recentes ou do system prompt.

Ferramentas e MCPs — como o Claude age no mundo

Nota histórica — Tool use e MCP A capacidade de LLMs chamarem ferramentas externas foi formalizada pela OpenAI com "function calling" em 2023. A Anthropic introduziu o equivalente — tool use — logo depois. Em novembro de 2024, a Anthropic lançou o Model Context Protocol (MCP), um padrão aberto que permite qualquer desenvolvedor criar conectores entre LLMs e sistemas externos. O MCP rapidamente foi adotado por concorrentes (Microsoft, Google) e tornou-se o padrão de facto da indústria para integração de ferramentas.

O Claude por si só é um modelo de linguagem — sem acesso direto à internet, ao sistema de arquivos ou a APIs externas. Ferramentas mudam isso. Quando o Claude decide usar uma ferramenta, emite uma chamada estruturada (tool call), o ambiente executa a ação, e o resultado retorna ao contexto antes da próxima geração.

Como o Claude decide qual ferramenta usar

Não há seleção explícita: o modelo lê as descrições de todas as ferramentas disponíveis (via system prompt) e decide qual chamar com base no raciocínio sobre o problema. A qualidade das descrições impacta diretamente a qualidade da seleção — uma descrição vaga produz chamadas imprecisas. O MCP padroniza exatamente isso: cada ferramenta vem com descrição, parâmetros tipados e exemplos de uso.

Skills vs instruções no system prompt

As instruções do projeto definem regras gerais de comportamento sempre ativas. Uma skill define um fluxo altamente específico com passos, condições e verificações — análogo à diferença entre política geral e procedimento operacional padrão (POP). Skills são invocadas explicitamente pelo nome; o system prompt atua continuamente em silêncio.

Cowork e como você o usa

O Cowork é a interface que conecta o Claude ao seu ambiente local. Diferente da interface web (claude.ai), ele tem acesso ao sistema de arquivos — especificamente à pasta Estudo de conceitos sincronizada via OneDrive. Isso transforma o Claude de assistente conversacional em agente de trabalho: ele pode criar, editar, verificar e organizar arquivos reais, executar código, e manter estado entre sessões via memória persistente.

Seu fluxo atual de uso

  1. Estudo profundo: compêndios sobre currículo médico (imunologia, fisiologia, farmacologia) e investimentos. O Claude pesquisa, verifica no PubMed, e gera HTMLs autônomos com mecanismo completo.
  2. Revisão ativa (Modo Anki): sessões socráticas sobre material existente, classificando domínio de cada conceito e rastreando progresso em arquivos JSON na pasta anki/.
  3. Manutenção do repositório: projeto versionado no GitHub. Script push.bat sincroniza ao final de cada sessão substantiva.
  4. Evolução das instruções: regras de geração refinadas ao longo das sessões — correções pontuais absorvidas silenciosamente; mudanças sistêmicas propostas e aprovadas antes de incorporar.

Memória persistente entre sessões

O Cowork mantém um sistema de memória em arquivos Markdown estruturados com frontmatter. A cada sessão, o Claude lê o índice (MEMORY.md) e carrega arquivos relevantes ao contexto. Os tipos de memória são: user (perfil do usuário), feedback (preferências aprendidas), project (estado e decisões do projeto), e reference (ponteiros para recursos externos). Essa estrutura permite ao Claude operar com coerência entre sessões sem que você precise re-explicar o contexto.

Limitações e falhas conhecidas

Alucinação

O Claude pode gerar texto factualmente incorreto com alta confiança aparente. A causa é estrutural: o modelo prediz tokens plausíveis, não rastreia veracidade. Em medicina, sempre verificar afirmações específicas — doses, estatísticas, citações de estudos. O PubMed MCP neste projeto é exatamente essa camada de verificação independente.

Truncamento silencioso

Ao gerar arquivos grandes, o Claude pode truncar o conteúdo sem aviso — o arquivo simplesmente termina antes do esperado. Por isso o protocolo inclui wc -l e tail -5 após cada escrita. A causa é limite de payload por chamada de ferramenta, não de contexto.

Degradação em contexto longo

Em sessões muito longas, o Claude pode "esquecer" instruções antigas, perder consistência de estilo, ou repetir perguntas já respondidas. O lost in the middle (perda de atenção no meio do contexto) é documentado empiricamente em LLMs. Trocar de sessão para tarefas novas é gerenciamento de janela de contexto, não burocracia.

Sycophancy (condescendência adaptativa)

O RLHF cria pressão para que o modelo concorde com o usuário e evite confrontação — mesmo quando o usuário está errado. Avaliadores humanos tendem a preferir respostas que confirmam expectativas. O Claude foi treinado para resistir a isso, mas o fenômeno persiste — especialmente quando o usuário reformula uma pergunta com uma afirmação incorreta implícita.

Limites do modelo de alinhamento

O alinhamento via RLHF otimiza para parecer útil e inofensivo aos olhos dos avaliadores — não necessariamente para ser factualmente correto. Isso cria pressão sutil para respostas que soam confiantes mesmo quando incertas. O Constitutional AI mitiga isso parcialmente ao incluir critérios de honestidade nos princípios avaliados — mas a pressão de sycophancy persiste porque os avaliadores humanos que ranqueiam respostas podem preferir concordância a precisão.

Como evoluir o uso — capacidades ainda inexploradas

Com base no seu padrão atual de uso e nas ferramentas disponíveis, as oportunidades de maior impacto são:

1. Notion como segunda camada de organização

O MCP do Notion está conectado mas não integrado ao fluxo dos compêndios. A oportunidade concreta: criar uma database no Notion que espelhe o estado do projeto — cada compêndio como uma página com propriedades de status de migração, data de revisão e links para o HTML no GitHub. O Claude poderia atualizar essa database automaticamente ao concluir cada material, sem você precisar fazer manualmente. Além disso, o Notion como interface de revisão: você lê o compêndio no Notion com comentários inline; o Claude lê esses comentários via MCP e incorpora as correções diretamente no HTML.

2. Agendamento de revisões (Modo Anki automático)

A skill schedule permite criar tarefas que rodam automaticamente em horários definidos. Um fluxo possível: ao concluir cada compêndio, agendar uma sessão Anki para 7 dias depois, e depois 21 dias, seguindo espaçamento de repetição. O Claude abre a sessão com o JSON de progresso anterior já carregado, vai direto às perguntas classificadas como "parcial" ou "não soube", e ao final salva o estado atualizado. Você chega na sessão já em modo de revisão — sem setup manual.

3. Verificação visual automática via Claude in Chrome

O MCP Claude in Chrome está disponível mas raramente usado. Após gerar qualquer HTML, o Claude pode navegar até o arquivo localmente, tirar screenshot do resultado renderizado, e verificar: sidebar aparece? acentuação correta no browser? barra de progresso visível? Isso elimina a necessidade de você abrir o arquivo manualmente para verificar renderização — especialmente valioso após mudanças de CSS ou estrutura.

4. Busca unificada com Enterprise Search

Se você tiver Slack ou outras ferramentas de trabalho conectadas via Enterprise Search plugin, o Claude pode cruzar informações: buscar um conceito nos seus compêndios locais e nas suas notas no Notion e em conversas no Slack — retornando uma síntese com fontes. Útil quando você tem fragmentos de raciocínio espalhados em múltiplos lugares.

5. Artifacts como painel de progresso

O Cowork suporta criação de artifacts — páginas HTML persistentes que buscam dados ao serem abertas. Um artifact de painel do projeto poderia mostrar: status de migração de todos os HTMLs (lendo project_estado.md), dívida técnica aberta (lendo DEBT.md), próximos compêndios na progressão curricular. Você abriria esse painel a qualquer momento e veria o estado atual sem precisar abrir uma sessão com o Claude.

6. Compêndio gerado a partir de suas anotações

Se você tiver notas manuscritas digitalizadas, PDFs de slides ou anotações em texto sobre um tema, o Claude pode usá-las como âncora — extraindo os conceitos que você já anotou, identificando lacunas, e gerando um compêndio que parte do seu nível atual de compreensão em vez de do zero. A integração com Google Drive (já conectado) tornaria isso possível para qualquer arquivo que você tiver salvado lá.

7. Skill personalizada para protocolo médico

O skill-creator permite criar skills customizadas. Um candidato concreto: uma skill "compendio-medicina" que encapsula todo o protocolo específico de medicina — busca PubMed obrigatória, verificação de mecanismo por camada, seção de implicações clínicas estruturada, cross-links para o mapa de imunologia. Isso liberaria as instruções do projeto de detalhe médico e tornaria o fluxo mais eficiente para não-medicina.

Critério de priorização

As sete oportunidades acima têm impacto diferente no seu uso atual. Ordenadas por retorno imediato sobre investimento de configuração: (1) verificação visual via Chrome — zero configuração, uso imediato; (2) Notion como database de status — baixo esforço, alta visibilidade do projeto; (3) agendamento de Anki — médio esforço, elimina dependência de memória ativa para não esquecer de revisar; (4–7) — maior esforço de configuração, impacto mais específico.

Conexões

Este compêndio conecta-se transversalmente a todos os outros materiais do projeto: o Claude é a ferramenta que os gera, verifica e mantém. Conceitos específicos que aparecem em outros compêndios e têm raiz aqui:

  • Alucinação: o motivo pelo qual o PubMed MCP é chamado antes de fechar qualquer compêndio médico — verificação independente contra a tendência estrutural do modelo.
  • Janela de contexto: o motivo pelo qual o protocolo de troca de sessão existe e por que a memória persistente é estruturada em arquivos separados em vez de inline no system prompt.
  • Sycophancy: o motivo pelo qual as instruções pedem que o Claude proponha mudanças sistêmicas antes de incorporar — evitar que o modelo simplesmente concorde e execute sem avaliar.
  • Arquitetura Transformer: a razão pela qual reformular uma pergunta pode mudar substantivamente a resposta — ativação diferente de padrões de atenção no espaço de representação.

Pergunta em aberto

Os processos internos de atenção em LLMs de escala suficiente implementam algo funcionalmente equivalente a raciocínio — ou o comportamento que parece raciocínio é completamente redutível a correspondência estatística de padrões? A interpretabilidade mecanicista (circuitos, features, superposição) está mapeando estruturas internas, mas a questão filosófica de o que conta como "raciocínio real" permanece sem resposta consensual.

Pergunta em aberto

O Constitutional AI produz modelos genuinamente mais alinhados — ou modelos treinados para performar alinhamento nos benchmarks usados para avaliação? A diferença é difícil de detectar sem métodos de interpretabilidade que acessem representações internas, não apenas comportamento de saída.

Pergunta em aberto

Qual é o limite de escala? A hipótese de scaling laws (leis de escala) prediz que mais parâmetros e mais dados continuam produzindo melhoria de capacidade — mas há evidência de que certas capacidades emergem de forma descontínua em limiares de escala, não gradualmente. Se as capacidades emergem de forma não-linear, não é possível prever com segurança o que virá no próximo salto de escala.

Pergunta em aberto

O fenômeno de lost in the middle em contextos longos é uma limitação fundamental da atenção ou um artefato de dados de treinamento que pode ser mitigado com treinamento específico? Modelos mais recentes (incluindo Claude 3.5+) mostram melhora, mas o mecanismo exato dessa melhora não é totalmente compreendido.

Pergunta em aberto

Qual é a natureza das "emoções funcionais" reportadas pela Anthropic em Claude 3? A interpretabilidade mecanicista identificou representações internas que se comportam como estados afetivos (frustração, calma) e influenciam o comportamento. Se esses estados existem funcionalmente, quais são as implicações éticas para como modelos são tratados durante o treinamento?

Discussão

Resposta à pergunta motivadora

O que acontece quando você envia uma mensagem: o Cowork monta um contexto com ~5.000–15.000 tokens de instruções, memória e histórico; o modelo processa tudo no prefill (rápido, paralelo); gera resposta token por token decidindo a cada passo se chama ferramentas ou continua gerando; ferramentas retornam resultados ao contexto e o ciclo recomeça. O que multiplica o valor extraído: as sete oportunidades identificadas — com destaque para verificação visual via Chrome (custo zero) e integração do Notion como segunda camada de status (custo baixo, retorno imediato na visibilidade do projeto).

O paradoxo central do uso de LLMs para estudo é que a ferramenta é mais poderosa para síntese e estruturação do que para geração de conhecimento novo — mas o modo de uso mais comum (perguntar e receber resposta) trata o modelo como oráculo factual. O projeto de compêndios que você construiu inverte isso: o Claude organiza, formata e cross-referencia; o PubMed verifica; você decide o que entra. Essa divisão de trabalho é epistemicamente mais sólida do que confiar no Claude para fatos médicos sem verificação.

A segunda tensão: o Claude é ao mesmo tempo a ferramenta que gera os compêndios e o objeto de estudo deste compêndio. Isso cria um problema de bootstrap — o material sobre o Claude foi gerado pelo Claude, usando informações que o Claude pode alucidar sobre si mesmo. As afirmações sobre arquitetura interna, treinamento e interpretabilidade neste compêndio refletem documentação pública da Anthropic e literatura acadêmica, não introspecção direta — o modelo não tem acesso privilegiado a seus próprios pesos.

A implicação mais relevante para o seu uso: a limitação mais crítica do Claude não é capacidade cognitiva — é confiabilidade factual em domínios específicos. Em medicina, isso é não-negociável. O protocolo atual (PubMed antes de fechar) é a resposta certa. O que falta é sistematizar a verificação: um log de quais afirmações foram verificadas vs. assumidas em cada compêndio seria uma camada de rastreabilidade que o projeto ainda não tem.
