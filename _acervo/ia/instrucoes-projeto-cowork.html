Fundamentos necessários

O que é o system prompt e por que ele importa

Toda interação com o Claude começa com um system prompt — um bloco de texto inserido antes de qualquer mensagem do usuário que estabelece o contexto, os objetivos, os limites e a personalidade com que o modelo vai operar. No Cowork, as instruções de projeto são o mecanismo pelo qual o usuário escreve e controla esse system prompt para uma pasta específica.

O system prompt não é um menu de opções — é uma especificação comportamental. O modelo o lê uma vez a cada nova sessão, junto com as mensagens do usuário, e tenta satisfazer simultaneamente todos os objetivos nele declarados. Isso tem consequências diretas: instruções contraditórias criam comportamento instável; instruções vagas são resolvidas pelo julgamento do modelo (que pode não coincidir com o do usuário); instruções precisas reduzem o espaço de interpretação e produzem saídas mais previsíveis.

Atenção como recurso escasso

Modelos de linguagem processam texto por meio de mecanismos de atenção — cada token influencia e é influenciado por outros tokens na janela de contexto. Em termos práticos, isso significa que o conteúdo colocado no início do contexto (início do system prompt, início de uma seção longa) recebe peso de atenção maior do que conteúdo no meio. Essa propriedade, documentada empiricamente sob o nome de lost in the middle, explica por que instruções críticas enterradas no parágrafo 15 de um system prompt de 20 parágrafos tendem a ser seguidas com menos fidelidade (LIU et al., 2023).

Tokens, janela e custo cognitivo

Cada caractere do system prompt consome tokens da janela de contexto e, indiretamente, do orçamento de atenção disponível para a conversa. Um system prompt de 10.000 tokens é viável tecnicamente (a janela do Claude 3 Sonnet é de 200.000 tokens), mas enfrenta dois problemas práticos: (1) a atenção se dilui — regras enterradas em textos longos são processadas com menor peso relativo; (2) o carregamento aumenta a latência de cada resposta, pois todo o prefill é reprocessado. A tensão entre completude e eficiência é inerente ao design de instruções.

Generalização vs. especificidade

O Claude foi treinado para generalizar: diante de uma situação não prevista nas instruções, ele infere o comportamento mais plausível dado o contexto. Isso é uma funcionalidade, não um bug — significa que não é necessário especificar todos os casos possíveis. Mas a generalização também significa que instruções ambíguas serão resolvidas por inferência, e a inferência do modelo pode divergir do que o usuário pretendia. A arte de escrever instruções de projeto está em saber o que especificar explicitamente e o que confiar à generalização.

Conceitos-chave

System Prompt

Texto inserido antes de qualquer mensagem do usuário; define o comportamento base do modelo para a sessão.

Instrução de Projeto

Conteúdo das configurações de projeto no Cowork; é injetado como system prompt em toda sessão da pasta.

Especificidade de Instrução

Grau em que a instrução delimita o espaço de respostas possíveis — do genérico ao prescritivo.

Lost in the Middle

Fenômeno pelo qual conteúdo posicionado no meio de contextos longos recebe menos atenção efetiva do modelo.

Grounding

Ancoragem do modelo em fatos, formatos ou restrições concretas; oposto de instrução abstrata.

Drift Instrucional

Degradação gradual da aderência às instruções ao longo de uma sessão longa, à medida que o contexto acumula tokens.

O que são instruções de projeto

Contexto histórico A ideia de condicionar o comportamento de um sistema de IA por meio de instruções textuais pré-fixadas remonta aos primeiros modelos de chat baseados em transformers (2018–2020), mas ganhou precisão prática com o surgimento de modelos instruídos (instruction-tuned) como o InstructGPT em 2022 e os modelos Claude em 2023. O Cowork formaliza essa capacidade como "instruções de projeto" — um system prompt persistente por pasta, que o usuário controla sem precisar editar código.

No Cowork, as instruções de projeto são um bloco de texto livre que o usuário configura em cada pasta de trabalho. A cada nova sessão naquela pasta, esse bloco é inserido automaticamente no início do contexto, antes de qualquer mensagem. O Claude as lê como especificação do "contrato" com o usuário para aquela pasta: o que fazer, como fazer, o que nunca fazer, qual formato adotar, qual vocabulário usar, que suposições carregar.

Definição formal

Instrução de projeto é um system prompt controlado pelo usuário que persiste entre sessões de uma pasta do Cowork. Ele estabelece objetivos, formato de saída, restrições comportamentais e contexto de domínio antes de qualquer turno de conversa. O modelo o lê em cada sessão junto com as mensagens subsequentes e tenta satisfazer todos os seus requisitos simultaneamente.

A diferença entre uma instrução de projeto e um prompt ad hoc enviado na conversa é a persistência e a posição no contexto. System prompts são processados antes de qualquer mensagem do usuário e recebem peso de atenção inicial elevado. Prompts enviados na conversa competem com o histórico de mensagens e tendem a ser "diluídos" à medida que a sessão cresce.

O que as instruções podem controlar

As instruções de projeto podem influenciar quatro dimensões do comportamento do modelo:

Formato e estilo de saída — extensão das respostas, uso de markdown, tom (formal, técnico, conversacional), idioma, estrutura de entrega (prosa vs. listas, headers vs. sem headers). O modelo segue especificações de formato com alta fidelidade quando são concretas e exemplificadas.

Domínio e contexto — área de conhecimento relevante, vocabulário esperado, nível de profundidade, referências canônicas do campo. Isso reduz o espaço de interpretação quando uma pergunta é ambígua: o modelo resolve a ambiguidade usando o domínio declarado nas instruções.

Fluxo de trabalho — sequência de etapas para tarefas recorrentes, critérios de qualidade, regras de verificação, protocolos de entrega. Quanto mais o usuário tem um workflow repetível, mais o modelo pode segui-lo automaticamente — sem precisar ser lembrado a cada sessão.

Restrições e limites — o que nunca fazer, o que confirmar antes de executar, que tipos de conteúdo não produzir, quando pedir esclarecimentos. Restrições claras reduzem a necessidade de correções posteriores.

Limites do modelo

Instruções de projeto não sobrescrevem os valores e treinamentos fundamentais do Claude. Regras de segurança, recusa de conteúdo prejudicial e princípios éticos do modelo têm precedência sobre o system prompt. Além disso, o modelo não é determinístico — instrução precisa reduz variabilidade, mas não a elimina. Em sessões muito longas, o drift instrucional pode degradar a aderência mesmo com instruções bem escritas.

Anatomia de boas instruções

O princípio da especificidade progressiva

Boas instruções operam em três níveis de especificidade, do mais amplo ao mais prescritivo. O erro mais comum é usar apenas um nível — instruções puramente abstratas (que o modelo generaliza de forma imprevisível) ou puramente prescritivas (que não escalam para casos não previstos).

Nível 1 — Objetivo e contexto: por que este projeto existe, qual problema ele resolve, para quem e em que circunstâncias. Essa informação âncora a generalização do modelo: quando encontrar um caso não coberto explicitamente, ele inferirá com base no objetivo declarado.

Nível 2 — Formato e padrões: como a saída deve ser estruturada, que convenções usar, que tom adotar. Exemplos concretos valem mais do que descrições abstratas de formato.

Nível 3 — Regras e restrições específicas: o que fazer em casos particulares, o que nunca fazer, quando pedir confirmação antes de agir. Este nível é prescritivo e deve ser reservado para situações que a generalização resolveria errado.

Exemplo bem estruturado

Objetivo: "Este projeto é para elaboração de laudos técnicos em perícia contábil judicial. O leitor é o juiz — formação jurídica, sem familiaridade com contabilidade avançada."

Formato: "Cada laudo começa com síntese executiva de no máximo 3 parágrafos. Termos técnicos são definidos na primeira ocorrência. Conclusões são numeradas e marcadas com grau de certeza (definitivo / provável / inconclusivo)."

Restrições: "Nunca emitir opinião sobre mérito jurídico — apenas sobre fatos contábeis. Antes de calcular valores retroativos, confirmar o período e a taxa de correção a usar."

Instrução orientada a mecanismo, não a resultado

Uma instrução que descreve o mecanismo de raciocínio (como pensar sobre o problema) é mais robusta do que uma que descreve apenas o resultado desejado (o que produzir). Isso porque o mecanismo se generaliza para casos novos; o resultado, não.

Fraco — orientado a resultado

"Produza relatórios completos e bem formatados."

Forte — orientado a mecanismo

"Antes de escrever qualquer relatório: (1) identifique a decisão que o leitor precisa tomar; (2) estruture as evidências do mais para o menos relevante para essa decisão; (3) termine com recomendação explícita ou com os critérios que faltam para tomá-la."

Exemplos como especificação

O Claude generaliza a partir de exemplos com mais fidelidade do que a partir de descrições abstratas. Isso é consequência direta do treinamento por few-shot learning: um ou dois exemplos de formato desejado ancoram o modelo melhor do que um parágrafo explicando o formato. Sempre que uma convenção for não-óbvia ou crítica, incluir um exemplo — não apenas uma descrição.

Princípio

Mostre, não apenas descreva. Para formatos, nomenclaturas, estruturas de entrega e níveis de detalhe: inclua pelo menos um exemplo representativo nas instruções. O modelo replica padrões demonstrados com maior precisão do que padrões narrados.

Elementos estruturais de instruções eficazes

Cabeçalho de contexto

O bloco inicial — idealmente o primeiro parágrafo — deve estabelecer o contexto do projeto em linguagem direta: o que é, para quem é, qual o objetivo central. Isso cria o frame de interpretação que o modelo usará para resolver ambiguidades ao longo de toda a sessão. Posicionar informação crítica no início não é apenas boa prática de leitura humana — é otimização para a distribuição de atenção do modelo.

Formato único ou formato principal

Projetos com um formato de saída dominante (um tipo de documento, uma estrutura recorrente) se beneficiam de declarar esse formato explicitamente e referenciá-lo como padrão. Quando há variações, descrevê-las como desvios do padrão (não como alternativas igualmente válidas) facilita a inferência do modelo em casos limítrofes.

Fluxo de trabalho sequencial

Para projetos com etapas recorrentes (pesquisa → rascunho → revisão → entrega, por exemplo), declarar a sequência explicitamente reduz a probabilidade de etapas serem puladas ou reordenadas. O ponto crítico é indicar não apenas o que fazer em cada etapa, mas quando passar para a próxima — critério de conclusão de etapa.

Hierarquia de prioridades

Quando houver risco de conflito entre objetivos (velocidade vs. completude, brevidade vs. precisão, formalidade vs. acessibilidade), as instruções devem declarar explicitamente qual prevalece. O Claude resolverá conflitos implícitos por inferência — e a inferência pode não coincidir com a prioridade real do usuário.

Protocolo de incerteza

Definir o que o modelo deve fazer quando não sabe algo é tão importante quanto definir o que deve fazer quando sabe. As três opções canônicas são: (a) pedir esclarecimento antes de agir; (b) declarar a incerteza e agir assim mesmo; (c) recusar a tarefa até receber informação adicional. A escolha depende do domínio e do custo de erro — um laudo pericial exige (a); um rascunho de email pode aceitar (b).

Glossário e convenções de nomenclatura

Em projetos de nicho (especialidade médica, área jurídica, terminologia interna de uma empresa), definir os termos específicos do domínio e a nomenclatura esperada elimina uma fonte relevante de variabilidade. O modelo conhece múltiplos usos do mesmo termo em contextos diferentes — o glossário nas instruções força a escolha correta para aquele projeto.

Separação entre regras permanentes e contexto variável

Uma instrução de projeto bem estruturada distingue o que nunca muda (regras, formato, valores do projeto) do que pode variar por sessão (contexto específico, material de referência, objetivo pontual). O que varia por sessão deve ser enviado na mensagem, não embutido nas instruções — para não torná-las obsoletas a cada novo caso.

Heurística de separação

Se você se pega abrindo as configurações do projeto para alterar algo a cada nova tarefa, isso é sinal de que aquele conteúdo pertence à mensagem, não às instruções. Instruções de projeto são para o que é verdadeiro em qualquer sessão do projeto.

Mecanismo cognitivo: como o Claude processa instruções

Por trás do processamento O entendimento de como modelos de linguagem processam system prompts evoluiu significativamente entre 2022 e 2025. Trabalhos de interpretabilidade mecanicista (Anthropic, EleutherAI, DeepMind) revelaram que a atenção não é uniforme — há "cabeças" de atenção especializadas em diferentes tipos de relação semântica. Isso explica por que certas formulações de instrução são sistematicamente mais eficazes do que outras, independentemente do conteúdo.

Processamento sequencial com bias posicional

O transformer processa o contexto em paralelo via atenção multi-cabeça, mas mantém codificação posicional que preserva a ordem dos tokens. Em termos práticos, tokens no início do contexto estabelecem um "frame" que influencia a interpretação de tokens subsequentes — análogo ao efeito de ancoragem em cognição humana. Instruções críticas posicionadas no início são processadas com mais peso relativo do que as mesmas instruções no meio ou no fim.

Resolução de conflitos implícitos

Quando duas instruções ou uma instrução e uma mensagem do usuário conflitam, o modelo aplica uma hierarquia implícita aprendida durante o treinamento: (1) princípios de segurança e valores fundamentais; (2) system prompt; (3) mensagem do usuário; (4) inferência a partir do contexto geral. Isso significa que a mensagem do usuário pode sobrescrever instrução de projeto em casos ambíguos — especialmente se a instrução não for explícita o suficiente para que o modelo a identifique como regra a preservar.

Consequência prática

Regras que não devem ser sobrescritas por mensagens do usuário precisam ser formuladas como obrigações explícitas — não como preferências. "Sempre confirme antes de deletar" é mais resistente à sobrescrita do que "é preferível confirmar antes de deletar". O verbo modal importa.

Drift instrucional em sessões longas

À medida que uma sessão acumula tokens (mensagens longas, arquivos anexados, saídas extensas), o peso relativo do system prompt no contexto total diminui. Isso produz o drift instrucional: a aderência às instruções degrada progressivamente, não porque o modelo "esquece", mas porque a atenção disponível para o system prompt diminui proporcionalmente ao crescimento do contexto de mensagens. Projetos com sessões longas se beneficiam de instruções mais curtas e de reafirmação periódica de regras críticas.

Instrução negativa vs. positiva

Instruções negativas ("não faça X") são processadas com menor fidelidade do que instruções positivas ("faça Y") — fenômeno documentado em múltiplos estudos de ablação em modelos de linguagem. Isso ocorre porque processar uma negação requer que o modelo ative a representação do comportamento proibido antes de inibi-lo. A versão robusta de uma restrição formula o comportamento desejado positivamente: em vez de "não use bullet points", usar "escreva em prosa contínua".

Menos eficaz — instrução negativa

"Não seja prolixo. Não adicione disclaimers desnecessários. Não use linguagem corporativa."

Mais eficaz — instrução positiva equivalente

"Respostas diretas: responda exatamente o que foi perguntado, sem preâmbulos ou ressalvas não solicitadas. Tom: técnico e preciso, vocabulário do domínio sem jargão corporativo."

Ancoragem por exemplos vs. ancoragem por descrição

A distinção entre instrução descritiva e instrução por exemplo tem base na mecânica de atenção: exemplos criam padrões de ativação concretos que o modelo pode replicar diretamente; descrições abstratas exigem que o modelo infira o padrão. Para formatos de saída, a diferença de fidelidade entre "forneça um exemplo" e "descreva o formato" é substancial — exemplos bem escolhidos reduzem a variabilidade de output em 30–50% em benchmarks de formatação (BROWN et al., 2020).

Fluxo de trabalho: da primeira versão à instrução madura

Fase 1 — Rascunho mínimo viável

A primeira versão das instruções deve responder apenas três perguntas: (1) O que este projeto é? (2) Qual é o formato padrão de saída? (3) Há alguma restrição absoluta? Tudo o mais pode ser adicionado depois, quando o uso real revelar lacunas. Instruções iniciais excessivamente detalhadas tendem a conter regras especulativas — formuladas antes de o usuário saber o que o modelo fará por padrão — e essas regras muitas vezes são desnecessárias ou contraproducentes.

Fase 2 — Acumulação orientada por fricção

A forma mais eficiente de amadurecer instruções de projeto é iterar a partir da fricção real: toda vez que o modelo produzir algo inesperado ou indesejado, perguntar se o comportamento poderia ser prevenido por uma instrução. Se sim, adicionar a instrução. Se não (comportamento emergente de generalização ruim ou limitação do modelo), registrar para não criar expectativas falsas. Instruções que surgem de fricção real são quase sempre mais úteis do que instruções especulativas.

Fase 3 — Poda e consolidação

Instruções acumuladas ao longo de muitos meses tendem a desenvolver redundâncias, contradições e regras obsoletas. Uma revisão periódica com a pergunta "esta regra ainda é necessária?" elimina ruído que dilui a atenção disponível para o restante. O critério de poda é simples: se remover a regra não mudaria o comportamento do modelo (porque já está coberta por outra regra ou pela generalização padrão), ela deve ser removida.

Separação de estado mutável e imutável

Um dos erros estruturais mais comuns em projetos de longa duração é misturar nas instruções conteúdo que muda a cada sessão (status atual, lista de pendências, material de referência) com conteúdo permanente (regras, formato, objetivo). O conteúdo mutável embutido nas instruções exige que o usuário abra as configurações a cada sessão — e instrução desatualizada é pior do que instrução ausente porque ativamente desinforma o modelo. Estado mutável pertence a arquivos externos que o usuário carrega na mensagem, não às instruções de projeto.

Arquitetura de estado

Instruções de projeto → o que é sempre verdadeiro. Mensagem inicial → o que é verdadeiro nesta sessão. Arquivos anexados → material de referência específico desta tarefa. Essa separação mantém as instruções estáveis e a sessão flexível.

Versionamento e autocrítica

Projetos sérios se beneficiam de tratar as instruções como código: manter um arquivo de registro das mudanças e a razão de cada uma. Isso serve a dois propósitos — evitar regressões (readicionar regras que foram removidas por boas razões) e construir entendimento de quais instruções resolvem quais problemas. A autocrítica ao final de cada sessão é o mecanismo de fechar o loop: o que funcionou, o que não funcionou, o que as instruções precisam de ajuste.

Antipadrões frequentes

Antipadrão 1 — Instrução como lista de desejos

O sistema de instruções não é um canal para expressar preferências genéricas ("seja criativo", "seja preciso", "entenda o contexto"). O modelo já tenta maximizar essas qualidades por padrão — instrução que repete o comportamento padrão consome espaço sem adicionar valor. Instruções úteis especificam o que o modelo não faria por padrão ou como resolver casos ambíguos em que há múltiplas respostas razoáveis.

Antipadrão — preferência sem operacionalização

"Seja sempre preciso e detalhado nas suas respostas. Demonstre conhecimento profundo do assunto. Evite superficialidade."

Essas instruções não mudam nada — o modelo já tenta ser preciso. O que muda o comportamento é especificar o que "preciso" significa naquele projeto: "cite a fonte primária quando fizer afirmação empírica" ou "defina todo termo técnico na primeira ocorrência".

Antipadrão 2 — Regras sem critério de aplicação

Uma regra sem critério de quando se aplica é interpretada pelo modelo como regra universal — com frequência produzindo comportamento indesejado em casos que não eram o alvo. "Sempre peça confirmação antes de agir" é perigoso se não especificar que tipo de ação requer confirmação: o modelo pode parar para confirmar até tarefas triviais.

Antipadrão 3 — Instrução por exclusão cumulativa

Construir instruções como uma lista crescente de proibições ("não faça A, não faça B, não faça C...") é menos eficaz do que descrever o comportamento positivo desejado. Além da questão cognitiva de processar negações, listas de proibição tendem a crescer indefinidamente — cada novo problema gera uma nova proibição — enquanto o comportamento positivo desejado permanece implícito e impreciso.

Antipadrão 4 — Misturar domínios em um único projeto

Um projeto cujas instruções precisam cobrir comportamentos radicalmente diferentes para tarefas diferentes (laudos técnicos E emails informais E análises de código, por exemplo) está mal delimitado. A solução é criar pastas de projeto separadas com instruções específicas — não adicionar mais condicionais às instruções existentes. Instruções condicionais ("se a tarefa for X, faça A; se for Y, faça B") são processadas com menor fidelidade do que instruções incondicionais.

Antipadrão 5 — Instrução de projeto como substituto para contexto na mensagem

Instrução de projeto não é o lugar para informação que varia por tarefa. Usuários que descrevem o histórico de cada projeto, as preferências do cliente do momento ou os detalhes da tarefa corrente nas instruções de projeto estão usando o mecanismo errado — e produzindo instruções que ficam obsoletas a cada novo caso. Essas informações pertencem à mensagem, onde podem ser atualizadas sem tocar nas configurações.

Manutenção e evolução das instruções

Quando atualizar

Três eventos justificam atualização das instruções: (1) comportamento indesejado que se repete em múltiplas sessões — o que indica lacuna nas instruções, não acidente; (2) mudança no objetivo ou no formato padrão do projeto; (3) descoberta de que uma instrução existente está produzindo comportamento indesejado em casos secundários (efeito colateral de regra excessivamente ampla). Atualização por curiosidade ("e se eu tentar X?") é válida como experimento, mas o resultado deve ser avaliado antes de ser incorporado permanentemente.

Quando não atualizar

Comportamento inesperado em uma única sessão raramente justifica mudança nas instruções — pode ser artefato de sessão longa, material de referência incomum ou caso genuinamente fora do escopo do projeto. O critério para atualização é recorrência: o problema se repetiu em condições normais? Se não, a resposta é ajustar a mensagem da sessão, não as instruções permanentes.

Testando mudanças

Instruções são código: mudanças devem ser testadas antes de serem consideradas definitivas. A forma mais simples de teste é iniciar uma nova sessão com as instruções atualizadas e executar a tarefa que causou o problema original — verificar se foi resolvida, sem introduzir novos problemas. Mudanças complexas (reformulação de múltiplas seções) merecem teste em pelo menos dois ou três casos representativos do projeto antes de serem definitivas.

Documentação de decisões

O risco de manter instruções ao longo do tempo sem documentação é acumular regras sem contexto: semanas depois, o usuário não sabe mais por que aquela restrição existe e pode removê-la por engano. A solução é incluir comentários inline nas instruções — não para o modelo (que os lerá junto com tudo o mais), mas para o usuário que revisará as instruções no futuro. O formato natural é um arquivo auxiliar de changelog ou simplesmente uma seção de "histórico de decisões" no início das instruções.

Limites do refinamento

Existe um ponto de retornos decrescentes no refinamento de instruções. Após atingir um nível de especificidade adequado para o domínio do projeto, adições marginais tendem a aumentar a complexidade das instruções sem melhorar proporcionalmente a qualidade das saídas — e frequentemente introduzem novos conflitos. O sinal de que chegou nesse ponto é quando as iterações de ajuste resolvem um problema e criam outro. Nesse caso, a resposta é simplificar, não adicionar.

Conexões

As instruções de projeto são aplicação direta do conceito de system prompt e tool use — o mecanismo pelo qual o Claude recebe contexto de comportamento antes de interagir com o usuário. A compreensão do fenômeno de lost in the middle e do drift instrucional conecta-se diretamente à arquitetura de atenção descrita em arquitetura do modelo.

A distinção entre instrução positiva e instrução negativa tem paralelos em cognição humana: a psicologia da instrução (ironic process theory de Wegner, 1994) documenta que instruções negativas ativam representações do comportamento proibido antes de inibi-lo — com frequência aumentando a probabilidade do comportamento que tentam prevenir. O Claude exibe fenômeno análogo por razões arquiteturais diferentes, mas com efeito prático similar.

O ciclo de refinamento de instruções descrito neste compêndio — rascunho mínimo, acumulação por fricção, poda periódica — é estruturalmente idêntico ao ciclo de melhoria contínua em engenharia de software (refactoring): código mínimo funcional, expansão orientada por casos reais, simplificação periódica para eliminar acúmulo de débito técnico.

Perguntas em aberto

Questão 1

Qual é o efeito preciso da posição de uma instrução dentro do system prompt sobre sua taxa de aderência? Os estudos de lost in the middle (Liu et al., 2023) documentaram o fenômeno para recuperação de informação em contexto, mas a extensão para seguimento de instruções comportamentais permanece pouco estudada experimentalmente.

Questão 2

Existe um tamanho ótimo de instrução de projeto — após o qual a adição de mais conteúdo começa a degradar a aderência às regras existentes? A intuição do campo sugere que sim, mas os limiares documentados variam significativamente entre modelos e versões, e não há consenso sobre métricas de "saturação instrucional".

Questão 3

Como o modelo resolve conflitos implícitos entre system prompt e mensagem do usuário quando ambos são tecnicamente válidos? A hierarquia formal é conhecida (system prompt tem precedência), mas os mecanismos de atenção que determinam qual instrução "vence" em casos borderline não estão completamente mapeados pela interpretabilidade mecanicista atual.

Questão 4

Instruções escritas em linguagem natural estruturada (com headers, listas, numeração) são processadas com maior fidelidade do que instruções em prosa contínua? Estudos preliminares de prompt engineering sugerem vantagem do formato estruturado para modelos modernos, mas os resultados são inconsistentes entre diferentes tipos de tarefa.

Discussão

A pergunta motivadora deste compêndio — por que projetos com o mesmo Claude produzem resultados radicalmente diferentes — tem uma resposta técnica e uma resposta estrutural. A resposta técnica é mecanicista: a distribuição de atenção, o bias posicional e o processamento diferenciado de negações vs. afirmações explicam por que certas formulações de instrução são processadas com mais fidelidade. Mas a resposta mais importante é estrutural: instruções eficazes são eficazes porque refletem uma compreensão clara do projeto — do que o projeto é, do que não é, e de como resolver os casos ambíguos. Instruções ineficazes são ineficazes porque o usuário ainda não sabe exatamente o que quer, e nenhuma formulação salva uma especificação vaga.

Isso aponta para uma tensão fundamental: as instruções de projeto são escritas no momento em que o usuário ainda tem menos informação sobre o projeto do que terá depois de meses de uso. A solução não é escrever instruções perfeitas de início — é aceitar que as instruções são um artefato vivo, que matura junto com o projeto, e construir o hábito de revisão periódica. Projetos que tratam as instruções como configuração inicial e depois as esquecem tendem a acumular regras obsoletas que desorientam o modelo, ou a nunca desenvolver as instruções que resolveriam os problemas recorrentes.

O paradoxo final é que as melhores instruções de projeto são invisíveis na prática: quando estão bem calibradas, o modelo se comporta exatamente como o usuário esperaria — e o usuário deixa de notar as instruções porque nunca precisa corrigi-las. O esforço de refinamento fica escondido na fluidez das interações. Isso cria um problema de valorização: usuários que nunca investiram em instruções tendem a subestimar seu impacto, porque não têm como comparar. A forma de resolver esse viés é deliberada — testar, medir o número de correções por sessão, e observar como esse número diminui à medida que as instruções amadurecem.

Prompt Engineering Guide — DAIR.AI. Recurso de referência continuamente atualizado sobre técnicas de prompting para LLMs modernos, incluindo seções sobre system prompts e few-shot learning. Acesso aberto: promptingguide.ai.

Anthropic Prompt Engineering Documentation — Documentação oficial do Claude sobre boas práticas de prompting, incluindo exemplos de system prompts e guias de formatação. Acesso aberto: docs.anthropic.com.

"Language Models are Few-Shot Learners" — Brown et al. (2020). Artigo seminal do GPT-3 que demonstrou a eficácia de exemplos (few-shot) em comparação com descrições abstratas para orientar o comportamento de modelos de linguagem. Acesso aberto: arxiv.org/abs/2005.14165.
