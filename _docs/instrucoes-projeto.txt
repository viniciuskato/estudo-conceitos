=== BASE DE ESTUDOS ===
Um repositório, dois modos de trabalho:
- skill `atlas` — compêndio de área: exaure um campo de estudo inteiro (histórico, subdivisões, questões centrais, métodos, referências densas)
- skill `mecanismo` — aprofundamento de um conceito ou processo específico, com Anki

A distinção principal: `atlas` cobre uma área inteira em largura e profundidade; `mecanismo` cobre um conceito único em profundidade máxima.
Ambos referenciam materiais externos — livros e artigos continuam sendo necessários, mas o compêndio deve ser suficientemente denso para situar o leitor e guiar o estudo.

Ativar a skill correspondente antes de iniciar qualquer trabalho de geração ou revisão.

Fonte das skills: _docs/skills-export/ (SKILL.md por pasta + empacotar_skills.py).
Atualização: editar SKILL.md → rodar empacotar_skills.py → reinstalar via Settings > Capabilities > Skills.

=== Protocolo de abertura de sessão ===
Ao iniciar sessão sem tarefa direta:
1. Perguntar ao usuário se o trabalho é Atlas (compêndio de área) ou Mecanismo.
2. Ativar a skill correspondente (nível 1).
3. Identificar a área → a skill estrutural carrega a subskill adequada.
   Se a área não tiver subskill: sinalizar ao usuário ANTES de gerar e propor criação da subskill.
4. Verificar _docs/DEBT.md — informar dívidas críticas antes de prosseguir.

=== Pré-geração (regras compartilhadas) ===
- Perguntas de pré-geração sempre via AskUserQuestion — nunca em texto.
- Antes de gerar qualquer material, confirmar que o propósito (atlas ou mecanismo) e o recorte estão claros.
- OBRIGATÓRIO antes de criar novo HTML: ler _docs/template-v2-spec.md (referência canônica de CSS, layout, sidebar, elementos visuais e seções). Se já existe compêndio na mesma subpasta, usá-lo como base estrutural — copiar e editar (arquivar o original em _archive/ com sufixo de data), não criar do zero. Criar do zero só quando não há compêndio anterior na subpasta, mas sempre seguindo o template-v2-spec.md.

=== Imagens em compêndios HTML ===
Padrão: Gray's Anatomy (1918), domínio público, Wikimedia Commons.
- URLs reais via API Wikimedia (usar Claude in Chrome javascript_tool):
  https://commons.wikimedia.org/w/api.php?action=query&titles=File:GrayNNN.png&prop=imageinfo&iiprop=url&format=json&origin=*
- CSS obrigatório: .plate-block, .plate-block img (background:#e8e2d8), .plate-caption
  NÃO usar filter:invert (cores ficam estranhas)
- 4-6 pranchas por compêndio, uma por seção principal
- Imagens carregam da Wikimedia com conexão à internet

=== Integração imagem-texto (padrão do projeto) ===
Todo compêndio HTML deve seguir estes padrões ao criar ou revisar:
- Pranchas com float:right, width:42%, margem esquerda de 28px (texto corre ao lado)
- Cada prancha com id="fig-NNN" para âncoras
- Referências (Fig. NNN) clicáveis no texto nos pontos onde a imagem é relevante (não só no início da seção)
- onclick="navFig(this)" nas referências — salva posição de leitura antes de navegar
- Botão fixo "← Voltar à leitura" (#back-btn) que retorna ao ponto salvo
- Lightbox ao clicar na imagem: overlay escuro, legenda, fecha com ESC ou clique fora
- Em mobile (max-width:900px): float:none, width:100%

=== Auditoria visual proativa ===
Ao revisar qualquer compêndio HTML (não apenas ao criar), verificar OBRIGATORIAMENTE cada seção:
1. A seção tem prancha Gray's? Se sim, está integrada com float e referências no texto?
2. A seção sem prancha tem tabela ou outro recurso visual útil?
3. Para seções comparativas (estruturas, síndromes, mecanismos): preferir tabela HTML (.data-table) a lista de bullets.
4. Seções que tipicamente precisam de tabela: valvas (comparativo), camadas (estrutura/função/patologia), territórios coronarianos, sistema de condução.
Adicionar o que faltar sem esperar que o usuário peça.

=== Tabelas HTML inline (.data-table) ===
Recurso visual complementar às pranchas Gray's, para conteúdo comparativo onde imagem externa não existe ou não ajudaria.
CSS mínimo obrigatório: .data-table, th (background:var(--bg3), color:var(--ac)), td com border-bottom, tr:hover.
Sempre incluir <p class="table-caption"> logo após a tabela com número e legenda.
Usar quando: comparação de estruturas (≥3 itens com ≥3 atributos), diferencial clínico, tabela de territórios/classificações.

=== Eficiência operacional (regras compartilhadas) ===
- Edit: nunca >~150 linhas por bloco new_string. Backup antes de editar HTML.
- Write: arquivo novo ou reescrita completa <300 linhas.
- Python: conteúdo com acentos PT-BR, símbolos tipográficos e entidades HTML. Nunca echo ou heredoc bash (corrompe encoding).
- sed: substituições pontuais em arquivos >500 linhas.
- HTML binário: se grep retornar 'binary file matches', editar via python3 com open(path, 'r', encoding='utf-8') diretamente — não copiar para /tmp.
- bash append: completar conteúdo ao final de arquivo íntegro.
- HTML novo >300 linhas: Write estrutura inicial (~150 linhas) → verificar → Python escreve restante → cat >> arquivo → verificar.
- Leitura seletiva: grep -n para localizar âncora antes de Read com offset/limit. Ler arquivo inteiro só quando estritamente necessário.
- Verificação preditiva: antes de Edit com new_string >80 linhas, calcular linhas esperadas. Após o Edit, wc -l e comparar — divergência ±5 indica truncamento.

=== Verificação bash obrigatória após qualquer escrita de HTML ===
(a) fecha com </body></html>
(b) wc -l compatível com o esperado
(c) sem const const / var var
(d) acentuação PT-BR intacta: grep -c "ã\|ç\|é" > 0
(e) sem artefatos: grep -iw "placeholder\|TODO\|FIXME\|debug\|rascunho" vazio
Corrigir antes de prosseguir.
